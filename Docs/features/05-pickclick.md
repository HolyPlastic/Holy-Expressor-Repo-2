# PickClick

> **Status: ACTIVE** -- PickClick is stable and working as of 2026-04-13. The detailed history below documents prior failure modes that led to the original quarantine; that quarantine has been lifted. The Open Bugs list (below) predates the current working state and requires an audit pass against the live code — see Dev Log entry 2.

**Files:** `js/main_PICKCLICK.js` -- `armPickClick` (exposed as `Holy.PICKCLICK.arm`), `cancelPickClick` (exposed as `Holy.PICKCLICK.cancel`), `setVeilActive`, `parseEventData`, `escapeForEvalScript` | `jsx/Modules/host_PICKCLICK.jsx` -- `he_PC_armPickClick`, `he_PC_cancelPickClick`, `he_PC_buildSignature`, `he_PC_getCoarseSignature`, `he_PICK_LeafProp_Snapshot`, `he_PC_poll` / `he_PC_scheduleNext` (`app.scheduleTask` polling), dispatched events `com.holy.expressor.pickclick.resolve` / `com.holy.expressor.pickclick.cancel` / `com.holy.expressor.pickclick.trace`
**UI Section:** Main Panel -- PickClick button / overlay veil (`#pickClickVeil`)

PickClick is an interactive property picker intended to provide a pick-whip-like UX from the CEP panel. The user arms PickClick, a translucent overlay veil appears over the panel, and the system waits for the user to click a property in the AE timeline. Host-side ExtendScript polling detects the selection change, resolves the property to an expression-compatible path (via `he_GET_SelPath_Simple`), and dispatches the result back to CEP for use as an expression target. The feature has never reached a stable, fully trusted implementation. It was quarantined while feasibility was unresolved; the feature has since reached a stable working state (status updated 2026-04-13).

*For quarantine rules, see the root `CLAUDE.md` Rule 10.*

## 5.1 Intended Behavior

When working correctly, PickClick should:

1. **Arm** -- User clicks the PickClick button. The panel displays a translucent overlay veil indicating "pick mode" is active. Host-side polling begins via `app.scheduleTask` (repeating, not single-shot chain).
2. **Snapshot** -- On arm, the host takes a baseline snapshot of the current AE selection state (coarse signature + deep snapshot, "Class 5" hybrid polling).
3. **Poll** -- The host repeatedly compares the current selection against the baseline snapshot. When a change is detected that resolves to a valid leaf property, the host builds an expression path.
4. **Resolve** -- The resolved property path is dispatched to CEP via `ISO_ReportLine_dispatch`. The veil closes. The path is inserted into the expression editor as a target reference.
5. **Cancel** -- Clicking the veil cancels PickClick. A 10-second safety timeout on the CEP side and a matching `HE_PICKCLICK_MAX_POLL_TICKS` (40 ticks * 250ms = 10s) cap on the host side also cancel if no resolution occurs (added in Dev Log entry 4).
6. **Fallback** -- When `he_GET_SelPath_Simple` cannot produce a formal expression path (due to missing accessor mappings), the host builds a display-name breadcrumb (e.g., `Layer 1 > Layer Styles > Drop Shadow > Opacity`) as a human-readable fallback. This breadcrumb is NOT a valid expression path.

## 5.2 Historical Quarantine Context (lifted 2026-04-13)

PickClick was previously quarantined because no iteration had reached a stable, correct, or fully trusted implementation. Partial successes were documented under narrow conditions in early testing (`Info_1.md`) but resolution was unreliable in subsequent sessions. The feature has since reached a stable working state. The items below describe the historical failure modes that led to the quarantine and require reconciliation against the current code (see Dev Log entry 2).

### Why it was quarantined (historical)

**`he_U_getSelectedProps` ghost function (Comment-Drift Incident)** -- A selection helper function (`he_U_getSelectedProps`) existed in the codebase, was commented out during experimentation, and was later fully deleted. Multiple agents treated the commented-out code as executable logic, causing repeated debugging misdirection. The function's removal was a clarifying action, not a declaration of failure, but it left PickClick without its original selection mechanism. The active trusted selection path is `he_GET_SelPath_Simple`, which PickClick did not successfully route through during the 2026-01-14 sessions.

**Hybrid polling failures** -- The Class 5 hybrid polling approach (coarse signature + deep snapshot) was adopted during the 2026-02-01 session. After a host patch, signature changes and resolve events appeared for some supported properties. However, unsupported property types caused polling to continue indefinitely. No full mapping of which AE property types the pick arm should accept vs. reject was ever established.

**Container selection pathology** -- AE selection APIs frequently return groups, containers, or non-expression-capable nodes rather than leaf properties. Shape layer hierarchies are deep and structurally inconsistent. No stable container-to-leaf promotion rule was established. This caused ambiguous intent, stalled resolution, and infinite polling loops.

**Breadcrumb fallback limitations** -- When `he_GET_SelPath_Simple` fails to resolve, the host builds a display-name breadcrumb. This mitigates the user experience but does not produce a valid expression path. The breadcrumb is rename-sensitive and locale-unsafe. `he_GET_SelPath_Simple` still lacks formal accessor mappings for many property types, so the fallback triggers frequently.

**Veil CSS unresolved** -- CSS `color-mix()` does not render in the CEP Chromium environment. The precise dark-themed veil styling was not resolved. A duplicate `#pickClickVeil` element was found and removed from `index.html`, but the visual design of the veil remains incomplete.

**CEP event silence** -- Early attempts assumed CEP could react to AE timeline clicks directly. CEP listeners were inconsistent or silent under tested conditions. This was deprioritized (not formally disproven) in favor of host-side polling, but it means the entire architecture relies on the polling mechanism, which itself is unreliable.

### What is NOT proven

The failure analysis (`PickClick - Canonical Investigation & Failure Analysis.md`) explicitly notes that no document conclusively proves impossibility. PickClick partially worked during early testing. Later pessimism was formed without full visibility of the archival evidence in `Info_1.md`. The investigation concluded with feasibility explicitly unresolved, not closed.
